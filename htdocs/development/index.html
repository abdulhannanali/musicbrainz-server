The following diagram briefly illustrates the relationships between the various MusicBrainz components:

Starting with probably the most familiar part, MusicBrainz has a web site.  You're reading one of its pages right now, of course.  Using the MB web site, you can search, browse and edit the MusicBrainz database: your web browser (top left) sends HTTP requests to the MB server (right) which then accesses the database as appropriate, and returns a response (usually a web page showing the requested information) to your browser.

As well as human-readable web pages, the MusicBrainz server also provides a (machine-readable) RDF interface.  RDF is an XML-based language.  By sending RDF queries to the MB server, a client can query and add to the database.  The client constructs an RDF message, and sends it to the server; the server parses and processes the message, accesses the database as necessary, and returns a response to the client. 

Although any client can send RDF queries to the MusicBrainz server, the easiest and most reliable way to do so is by using the MusicBrainz client.  A program makes the appropriate calls to the client library; then the MB client then constructs the correct RDF query, sends it to the server, receives the server's response, and finally returns the appropriate data to the calling program.  Thus the program doesn't have to know anything about RDF or about how to contact the MusicBrainz server.

Whilst most client applications (e.g. media players) will probably use only the RDF interface (via the MB client library), sometimes it is useful to also use the web interface at the same time.  This is the approach taken by both the tagger and the CD lookup tool.  For example the tagger will read a digital audio file, generate a TRM signature (using the client library), look up that TRM on the server (using the client library) and show the results to the user.  Then, when the user chooses to identify the file, the tagger points its embedded web browser component to the MusicBrainz web site, where the user can perform various searches.  Finally when the user selects the correct track, the tagger associates the TRM with the selected track (using the client library again).

CVS Modules

mb_server
The MusicBrainz server software.  To make a working MusicBrainz server you'll also need various other pieces of software (such as Apache and Perl) and a copy of the MusicBrainz database (which isn't stored in CVS).
mb_client
The MusicBrainz client library. 
trm
The MusicBrainz trm application/library.  This project is both a library and an application, used to generate TRM IDs from WAV, MP3 and Ogg Vorbis files.
mb_tagger
The MusicBrainz Tagger.  You'll also need the client and the trm library and Borland C++ Builder 5.0 to build this project.
tunepimp
Easy to use library for lookup and tagger. Does all the heavy lifting of decoding and lookup so you can focus on your UI.
python-musicbrainz
Python wrappers for mb_client.  Lets you code a musicbrainz app from python.
